AdCampaign Dashboard — Technical Documentation

Complete reference for the frontend application, backend API server, shared libraries, and monorepo workspace structure.

React 18 + Vite 7 TypeScript 5.9 Tailwind CSS v4 pnpm Workspace Frontend-only auth

🗂Repository Structure

The project is a pnpm monorepo managed with workspaces. All packages live under three top-level directories: artifacts/, lib/, and scripts/.

workspace/ ├── artifacts/ # Deployable applications │ ├── ad-campaign-dashboard/ # ⭐ Main React dashboard (frontend-only) │ ├── api-server/ # Express 5 REST API (unused by dashboard) │ └── mockup-sandbox/ # Vite server for canvas component previews │ ├── lib/ # Shared libraries │ ├── api-spec/ # OpenAPI 3.1 spec + Orval codegen config │ ├── api-client-react/ # Generated React Query hooks │ ├── api-zod/ # Generated Zod schemas │ └── db/ # Drizzle ORM schema + PostgreSQL connection │ ├── scripts/ # Utility scripts (single workspace package) │ └── src/ │ └── *.ts # Individual runnable scripts │ ├── pnpm-workspace.yaml # Declares artifacts/*, lib/*, scripts/* ├── tsconfig.base.json # Shared TS options (composite, bundler resolution) ├── tsconfig.json # Root project references └── package.json # Root hoisted devDeps
The dashboard (ad-campaign-dashboard) is completely self-contained — it does NOT call the API server at runtime. All data lives in localStorage. The API server exists as a separate scalable backend for future integration.

🚀Entry Points

src/main.tsx

The React root. Wraps the entire app in <AppProvider> so every component can access global state via the context hook.

ReactDOM.createRoot(document.getElementById('root')!).render(
  <AppProvider>
    <App />
  </AppProvider>
);

src/App.tsx

The top-level router. Reads currentUser from context — if null, renders <Login />; otherwise renders <Dashboard />.

export default function App() {
  const { currentUser } = useAppContext();
  return currentUser ? <Dashboard /> : <Login />;
}

localStorage Sync

Two effects handle persistence:

// On mount: hydrate from localStorage
useEffect(() => {
  const stored = localStorage.getItem('ad_campaign_db');
  if (stored) { const data = JSON.parse(stored); setUsers(data.users); … }
}, []);

// On any state change: persist back
useEffect(() => {
  localStorage.setItem('ad_campaign_db', JSON.stringify({
    users, entries, pos, regions, products, activities, bills
  }));
}, [users, entries, pos, regions, products, activities, bills]);

Exposed Functions

login(id, pass)

Finds matching user by loginId + password. Sets currentUser. Returns true/false.

logout()

Clears currentUser state (does not wipe localStorage data).

addEntry(entry)

Creates a new Entry with auto-generated UUID, status 'pending'.

updateEntry(id, updates, editedBy)

Merges partial updates into an Entry, records the editor name.

updateEntryStatus(id, status, decidedBy)

Approve/reject an entry — sets status + decidedBy + decidedAt timestamp.

deleteEntry(id)

Removes an entry by ID.

addPO / updatePO / approvePO / rejectPO / lapsePO

Full CRUD for Purchase Orders including approval workflow.

addUser / updateUser / deleteUser

Full CRUD for users.

addBill / updateBill

Create and update billing records.

calcLiveSpent(filters)

Sums amount of approved entries matching optional po/region/zone/area/product/activity/vendorId filters.

calcPendingSpent(filters)

Same as above but for pending entries.

getVisiblePOs()

Returns POs filtered to what the currentUser's role/territory can see.

getVisiblePendingEntries()

Returns entries the current approver needs to act on.

getMyEntries()

Returns entries created by the current user.

getScopedEntries()

Returns all entries within the current user's territory scope.

toast(msg, type)

Shows a temporary notification (auto-dismisses after 3 seconds).

📄Pages

Login.tsx

A split-panel layout — left branded panel visible on large screens, right glassmorphism card form visible on all sizes.

  • Left: dark mesh-gradient background, tagline, feature bullets, role avatar row
  • Right: glass card with "Welcome back" heading, SVG-icon inputs, gradient sign-in button with spinner
  • 6 quick-login demo buttons (one per role) for development/demo access
  • Error displayed inline in a red banner below the heading

Dashboard.tsx

The shell that wraps all 15 tabs. Never unmounts — tab switching just swaps the rendered child.

  • Header: diagonal gradient (navy → mid-blue), logo, user avatar + name, pending-items amber pill, sign-out button
  • Tab navigation: horizontal scrollable bar with underline active indicator, emoji icon per tab, badge count on Approvals and PO Approvals
  • Breadcrumb strip: shows "Dashboard › [Active Tab]" beneath the nav
  • Tab visibility: controlled by role flags (canSeeSheet, canSeeApprovals, etc.) — tabs not visible to the current role are simply excluded from the array
  • Toast: <Toast> rendered at root level from toastMsg context state

🔐Authentication Flow

There is no server-side session. Auth is fully client-side using the users array in state (backed by localStorage).

⚠️ Passwords are stored in plain text in localStorage. This is intentional for a demo/prototype — in a production system, passwords must be hashed server-side.

💾localStorage Schema

All state is serialised to a single localStorage key: ad_campaign_db.

{
  "users":      User[],
  "entries":    Entry[],
  "pos":        PO[],
  "regions":    Region[],
  "products":   string[],
  "activities": string[],
  "bills":      Bill[]
}

The context reads this on mount and writes to it after any state change via a useEffect with all slices as dependencies. To reset the app to defaults, clear ad_campaign_db from browser DevTools → Application → Local Storage, or use the Settings tab.

🗃Database Layer — lib/db

Package: @workspace/db. Uses Drizzle ORM over a PostgreSQL pool provided by Replit's built-in database service.

lib/db/ ├── src/ │ ├── index.ts # Creates pg Pool from DATABASE_URL, exports db + schema │ └── schema/ │ └── index.ts # Barrel re-export of all Drizzle table definitions └── drizzle.config.ts # Drizzle Kit config for migrations

Codegen Pipeline

The lib/api-spec package owns an openapi.yaml spec. Running pnpm --filter @workspace/api-spec run codegen (via Orval) generates:

  • lib/api-client-react/src/generated/ — React Query hooks (e.g. useHealthCheck)
  • lib/api-zod/src/generated/ — Zod schemas (e.g. HealthCheckResponse)

🧪Mockup Sandbox

Package: @workspace/mockup-sandbox. A Vite dev server specifically for canvas component previews. It serves isolated component URLs at /preview/<component-name> so components can be embedded as iframes on the Replit canvas board without showing the full app.

This package is managed by the Replit workspace and is not deployed to production.

📜Scripts Package

Package: @workspace/scripts. Individual TypeScript utility scripts, each runnable via:

pnpm --filter @workspace/scripts run <script-name>

Scripts can import any workspace package (e.g., @workspace/db) by declaring it as a dependency. Useful for one-off migrations, data seeding, or admin tasks.
